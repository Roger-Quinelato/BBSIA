2.2 A decisão "100% offline" — mérito e risco

✓ Ponto Positivo

Processar documentos governamentais sensíveis localmente (sem chamadas a APIs de nuvem) é a decisão correta do ponto de vista de LGPD, soberania de dados e requisitos de segurança do setor público. O LIIA opera em ambiente de dados governamentais — vazamento para LLMs externos seria um passivo jurídico relevante.

⚠ Tensão com o PRD

O documento de infraestrutura especifica Cloud Híbrida (LNCC/RNP + cloud pública). Isso significa que a arquitetura-alvo permite componentes em nuvem desde que em infraestrutura soberana. O "100% offline" do protótipo atual corresponde ao modo on-premise da cloud híbrida, mas deixa de fora a capacidade de elasticidade horizontal que o PRD exige para suportar pico de 16.400 usuários/dia no Ano 5.

2.3 Avaliação do estágio real do projeto

O protótipo atual representa aproximadamente 30% do escopo técnico do Sprint 0 + Sprint 2 do PRD: ele entrega o núcleo de busca semântica e uma UI funcional, mas nada dos Sprints 3 (autenticação), 4 (frontend completo) ou da infraestrutura de dados persistente. Isso é esperado para um PoC/MVP de laboratório — mas deve ser comunicado claramente à governança do LIIA para evitar confusão sobre o grau de prontidão para produção.

§ 3

Engenharia de RAG

3.1 O que está bem — Pipeline Híbrido Dense + Sparse + RRF

✓ Decisão arquitetural acertada

A combinação Dense (FAISS + sentence-transformers) + Sparse (BM25) + Reciprocal Rank Fusion é o estado da arte para RAG em domínio especializado com vocabulário técnico. BM25 compensa a fraqueza do dense search em termos exatos (siglas, IDs, nomenclaturas legais), enquanto o dense recupera intenções semânticas. O RRF é matematicamente superior a score-averaging para fusão de rankings. Esta escolha está correta.

3.2 Problema Crítico: Modelo de Embedding Inadequado

🔴 Crítico — Impacto direto na qualidade de resposta

O modelo configurado é paraphrase-multilingual-mpnet-base-v2 (dimensão 768, ~110M parâmetros), um modelo genérico de similaridade semântica. Para português técnico/jurídico/governamental, o correto seria intfloat/multilingual-e5-large (dimensão 1024, ~560M parâmetros) ou BAAI/bge-m3, que foram treinados com dados de recuperação de informação (MTEB retrieval tasks). A diferença em qualidade de recuperação em domínio especializado é tipicamente de 8-15 pontos percentuais no MRR@10.

3.4 O Faithfulness Check é um Castelo de Areia

🔴 Crítico — Falsa segurança anti-alucinação

A função _faithfulness_check() em rag_engine.py detecta alucinações checando se a resposta contém [Fonte N] via regex. Um LLM pode fabricar afirmações completamente falsas e ainda assim citar fontes — o modelo pode gerar "O projeto custa R$50M [Fonte 1]" mesmo que Fonte 1 não diga isso. O que existe é um verificador de formato, não de fidelidade.

A solução robusta envolve verificação cruzada com NLI (Natural Language Inference):

# Substituto robusto usando NLI local via sentence-transformers
from sentence_transformers import CrossEncoder

NLI_MODEL = "cross-encoder/nli-deberta-v3-small"  # ~184MB, roda local

def faithfulness_nli(
    claim: str,
    context_chunks: list[str],
    threshold: float = 0.6,
) -> bool:
    """Retorna True se algum chunk suporta o claim via NLI."""
    nli = CrossEncoder(NLI_MODEL)
    pairs = [(chunk, claim) for chunk in context_chunks]
    scores = nli.predict(pairs, apply_softmax=True)
    # scores: [contradiction, neutral, entailment]
    entailment_scores = [s[2] for s in scores]
    return max(entailment_scores) >= threshold

3.5 Calibração do Threshold de Retrieval

O parâmetro MIN_DENSE_SCORE_PERCENT = 18 (equivale a 0.18 em escala 0-1) é extremamente baixo para um índice FAISS com Inner Product. Com embeddings normalizados (como é o caso), os scores variam de -1 a +1 e um threshold de 0.18 aceita correspondências muito fracas. O script calibrar_threshold.py usa o percentil 25 dos scores observados como sugestão — mas percentil 25 significa que 25% das suas queries de teste reais ficariam abaixo do threshold, o que não é aceitável.

⚠ Recomendação

Use percentil 50 (mediana) dos scores dense em queries relevantes como threshold base. Além disso, o script de calibração deve ser executado separando queries "in-scope" de queries "out-of-scope" (como a "receita de bolo") — o threshold deve maximizar a separação entre as duas distribuições, não usar um percentil fixo de todas as queries misturadas.

3.6 Ausências Relevantes no Pipeline RAG

  • ALTO
    Sem histórico de conversa

    Cada chamada a /chat é stateless. Perguntas de acompanhamento ("e no Ano 2?") não têm contexto da mensagem anterior. Implementar um conversation_id com histórico no payload é crítico para UX real.

  • ALTO
    Sem Query Expansion / HyDE

    Perguntas curtas ou ambíguas ("Quais os riscos?") prejudicam o retrieval. Hypothetical Document Embeddings (HyDE) — gerar um documento hipotético como query — melhora recall em 10-20% para queries vagas.

  • MÉDIO
    Re-ranker desabilitado por padrão

    ENABLE_RERANKER=false. O cross-encoder melhora precisão consideravelmente em documentos corporativos. Deve ser habilitado por padrão com modelo leve (cross-encoder/ms-marco-MiniLM-L-6-v2, apenas 66MB).

  • MÉDIO
    Sem busca por grafo / linhagem

    O PRD especifica busca por grafo e similaridade. Para documentos governamentais com hierarquias (lei → decreto → portaria), um grafo de referências cruzadas aumentaria drasticamente a relevância contextual.

  • BAIXO
    BM25 não persistido

    O índice BM25 (Counter + defaultdict) é reconstruído em memória a cada _load_resources(). Para corpora grandes (>100k chunks), isso adiciona latência de startup significativa.

4.2 Problema Crítico: Endpoints Síncronos + Biblioteca requests

🔴 Crítico — Bloqueio do Event Loop

Todos os endpoints em api.py são definidos com def (síncrono), e rag_engine.py usa a biblioteca requests para chamar o Ollama com streaming. No FastAPI, def endpoints rodam em um thread pool — isso é aceitável para operações rápidas, mas a chamada ao Ollama com timeout de 300s bloqueia um thread por 5 minutos. Com 10 usuários simultâneos, o pool se esgota. A solução é adotar async def + httpx.AsyncClient:

# ANTES (bloqueante — péssimo)
def query_ollama(prompt: str, model: str) -> str:
    with requests.post(
        f"{OLLAMA_URL}/api/generate",
        json={"model": model, "prompt": prompt, "stream": True},
        stream=True, timeout=(10, 300),
    ) as response: ...

# DEPOIS (não-bloqueante — correto)
import httpx
from fastapi.responses import StreamingResponse

async def query_ollama_stream(
    prompt: str,
    model: str,
    client: httpx.AsyncClient,
):
    async with client.stream(
        "POST", f"{OLLAMA_URL}/api/generate",
        json={"model": model, "prompt": prompt, "stream": True},
        timeout=httpx.Timeout(10.0, read=300.0),
    ) as response:
        async for line in response.aiter_lines():
            if line:
                data = json.loads(line)
                if piece := data.get("response"):
                    yield piece
                if data.get("done"):
                    break

# Endpoint com streaming real ao cliente
@app.post("/chat/stream")
async def chat_stream(payload: ChatRequest) -> StreamingResponse:
    chunks = await retrieve_chunks(payload.pergunta)
    context = build_context(chunks)
    prompt = build_prompt(payload.pergunta, context)

    async def generate():
        async with httpx.AsyncClient() as client:
            async for token in query_ollama_stream(prompt, payload.modelo, client):
                yield token

    return StreamingResponse(generate(), media_type="text/plain")

4.3 Race Condition no Cache Global

🔴 Crítico — Corrupção de estado em concorrência

O cache é um dicionário global mutable: _CACHE: dict = {}. A função reload_resources() chama _CACHE.clear(), que pode ser executada enquanto outra thread serve uma requisição que ainda está usando os objetos do cache. Se o reprocessamento roda e limpa o cache no meio de uma busca FAISS, a busca seguinte encontrará o índice descarregado e levantará FileNotFoundError. A solução correta usa threading.RLock com copy-on-write:

import threading

class IndexStore:
    """Thread-safe singleton para os recursos do índice."""

    def __init__(self):
        self._lock = threading.RLock()
        self._data: dict | None = None

    def get(self) -> dict:
        with self._lock:
            if self._data is None:
                self._data = _load_from_disk()
            return self._data

    def reload(self) -> None:
        new_data = _load_from_disk()   # carrega ANTES de adquirir o lock
        with self._lock:
            self._data = new_data    # troca atômica — zero downtime

index_store = IndexStore()

4.4 FAISS em RAM — Gargalo de Memória e Startup

A linha embeddings = index.reconstruct_n(0, index.ntotal) carrega todos os vetores do índice FAISS de volta para numpy arrays em RAM. Para 10.000 chunks × 768 dimensões × float32 = ~31MB. Para 100.000 chunks (escala do BBSIA real) = ~310MB apenas de vetores, mais o overhead do índice, modelo de embedding, modelo BM25 e re-ranker. Em um servidor com 8GB RAM, isso é viável, mas o startup fica lento e sem hot-reload.

ℹ Solução para Médio Prazo

Migrar para pgvector (extensão PostgreSQL) ou Qdrant (vector store dedicado, open-source, com persistência em disco e API REST). Ambos eliminam a necessidade de carregar tudo em RAM e suportam busca híbrida nativa com HNSW (mais rápido que FAISS Flat para corpora grandes).

4.5 Rate Limiting Quebra em Multi-Worker

O rate limiting usa defaultdict(deque) com threading.Lock — funciona em um único processo. Com uvicorn --workers 4, cada worker tem sua própria cópia do dicionário: um usuário pode fazer 4× o limite configurado. Para produção, o rate limiting deve ser centralizado em Redis (usando fastapi-limiter ou implementação manual com redis-py + Lua scripts para atomicidade).

4.6 Classificador de Artigo — Heurísticas Frágeis

A extração de título em classificador_artigo.py usa tamanho de fonte como proxy de hierarquia. Isso funciona para PDFs com estilos consistentes, mas falha em: PDFs gerados por scan (fonte uniforme em toda a página), apresentações exportadas como PDF, documentos com capas gráficas e documentos que usam imagens como títulos. O resultado atual é qualidade_extracao="media" para a maioria dos documentos — o que é honesto, mas limita a qualidade das citações geradas pelo RAG.

ℹ Alternativa para Médio Prazo

Usar Docling (IBM, open-source) ou Unstructured.io para extração estruturada de PDFs. Ambos retornam hierarquia semântica (título, seção, parágrafo, tabela) com metadados de layout, são compatíveis com LangChain/LlamaIndex, e suportam OCR nativo via Tesseract.

4.7 Outros Achados Menores

  • ALTO
    Sem observabilidade

    Nenhuma instrumentação de métricas (Prometheus, OpenTelemetry). Impossível monitorar latência P95 de retrieval vs. geração, taxa de respostas sem citação, ou taxa de fallback para extractive. Crítico para um sistema RAG em produção.

  • ALTO
    Sem validação de qualidade RAG (RAGAS/DeepEval)

    A suite de 40 testes cobre funcionamento de módulos, não qualidade de retrieval. Não há benchmark de MRR@K, Faithfulness Score, Context Relevance ou Answer Relevance. Sem isso, é impossível saber se uma mudança de parâmetro (ex: alterar chunk size) melhora ou piora as respostas.

  • MÉDIO
    CATEGORIAS_DOCUMENTOS hardcoded no chunking

    O mapeamento de nomes de arquivo para área/assunto está hardcoded em Python. Qualquer novo documento não mapeado cai em area="geral", degradando a qualidade dos filtros e do retrieval. Deve ser movido para um arquivo de configuração externo ou para o próprio biblioteca.json.

  • BAIXO
    Ausência de paginação em /biblioteca

    O endpoint retorna todos os documentos sem paginação. Para corpora grandes, isso cria payloads pesados. Adicionar ?page=1&limit=20 com cursor-based pagination.

Conclusão

Diagnóstico Final

O BBSIA RAG é um protótipo tecnicamente sólido com escolhas de engenharia corretas no núcleo (busca híbrida, parent-child chunking, pipeline de extração em fases). O problema não é a qualidade do código escrito, mas o que ele não é ainda: não é o sistema descrito no PRD do LIIA, não é assíncrono, não é seguro para concorrência, não é monitorável, e não escala além de dezenas de usuários simultâneos.

A trajetória de maturação existe e é clara. A Fase 1 pode ser executada em 2-3 semanas e já entrega uma melhoria mensurável em qualidade de respostas e estabilidade. As Fases 2 e 3 constroem o sistema que o LIIA efetivamente precisa para cumprir a missão institucional de servir 575 entidades governamentais.

O ativo mais valioso que existe hoje é a arquitetura de pipeline bem pensada e testada. Refatorar não significa reescrever — significa crescer a partir de uma fundação que, na sua parte essencial, está correta.
